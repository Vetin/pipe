# Quality Rubric

Use this rubric when reviewing artifacts or scoring skill output.

| Area | Pass condition |
| --- | --- |
| Requirements | Requirements and acceptance criteria are testable and traceable. |
| Architecture | Module communication, security, failure, observability, and rollback are explicit. |
| Architecture delta | New, modified, removed, and unchanged behavior plus alternatives are explicit. |
| Technical design | Contracts, data model, errors, tests, ownership, dependency, migration/rollback, and decision traceability are actionable. |
| Slicing | Slices are small, ordered, TDD-ready, complexity-scored, ownership-aware, conflict-risk-aware, and linked to requirements and ACs. |
| Readiness analysis | Cross-artifact duplicate, ambiguity, underspecification, inconsistency, coverage, dependency, and constitution risks are checked before implementation. |
| Evidence | Raw red, green, verification, and review outputs exist in order. |
| Review | Findings link artifacts, reasoning, fix verification, and blocking status. |
| Memory | Feature-card and canonical docs explain motivation, implementation, tests, reviews, and evidence. |
