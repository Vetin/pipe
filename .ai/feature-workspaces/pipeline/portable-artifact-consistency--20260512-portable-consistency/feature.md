# Feature: Portable Artifact Consistency

## Intent

Fix the remaining drift after artifact-readability promotion: shared knowledge
must include the newest canonical feature, generated project profiles must be
portable, discovered-signal wording must not contradict canonical entries,
execution logs must use one structured slice-event schema, evidence manifests
must not call semantic labels hashes, and source readability must be guarded by
tests.

## Motivation

Future agents depend on checked-in feature memory. If that memory is stale,
machine-specific, or semantically ambiguous, pipeline runs can make incorrect
context decisions even when individual artifacts validate.

## Actors

- Codex agents using `.ai/knowledge` during context discovery.
- Maintainers reviewing pipeline artifacts through GitHub raw/diff views.
- Benchmark reviewers comparing hard and soft skill-quality scores.

## Goals

- Keep canonical and knowledge feature memory synchronized.
- Make checked-in knowledge portable across machines.
- Keep execution and evidence metadata unambiguous.
- Guard source readability with deterministic tests.

## Requirements

## Functional Requirements

- FR-001: Promotion and validation must keep `.ai/features/index.yaml`,
  `.ai/features/overview.md`, `.ai/knowledge/features-overview.md`, and
  `.ai/knowledge/project-index.yaml` synchronized.
- FR-002: Project profile output must not commit local absolute repository
  paths into `.ai/knowledge/project-index.yaml`.
- FR-003: `discovered-signals.md` and project overview must use
  `Canonical reason` for canonical entries and `Why not canonical` only for
  noncanonical signals.
- FR-004: Active `## Event Log` sections must use structured
  `event_type=slice_completed` and `event_type=slice_retry_completed` slice
  events.
- FR-005: Semantic completion labels must be stored as `change_label`, not
  `diff_hash`.
- FR-006: Source-controlled Python, YAML, Markdown, and `.gitignore` files must
  remain line-readable.
- FR-007: Worktree paths stored in feature/state artifacts must use one
  repository-relative convention.

## Non-Functional Requirements

- NFR-001: Validation errors must name the drifted artifact.
- NFR-002: Migration must preserve existing review and evidence history.
- NFR-003: Source-controlled artifacts must remain readable in normal Git diffs.

## Acceptance Criteria

- AC-001: Validation fails when a canonical feature key exists in
  `.ai/features/index.yaml` but not in `.ai/knowledge/features-overview.md` or
  project-index canonical features.
- AC-002: `init --profile-project` writes `project.root: .`.
- AC-003: Canonical catalog rows render `Canonical reason`, while lab/detected
  rows render `Why not canonical`.
- AC-004: Validation rejects unstructured `completed slice ...` events in
  `## Event Log`.
- AC-005: `complete-slice --diff-hash semantic-label` writes
  `change_label: semantic-label` and no `diff_hash`.
- AC-006: Tests reject non-generated Python source lines over 220 characters
  and source-controlled artifacts with collapsed newlines.
- AC-007: Feature and state worktree paths use `../worktrees/<name>`.

## Assumptions

- Existing canonical features are trusted memory and can be mechanically
  migrated for schema consistency.
- `../worktrees/<name>` remains the preferred external worktree convention.

## Open Questions

- Whether future work should split `featurectl.py` into modules is left for a
  dedicated maintainability feature.

## Non-Goals

- Splitting `featurectl.py` into modules in this pass.
- Removing showcase commands from `pipelinebench.py` in this pass.
- Changing evidence files generated by older completed features beyond metadata
  migration required for validation consistency.
