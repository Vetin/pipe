# Feature: Guardrail Status And Summary Polish

## Intent

Close the remaining review findings around public raw guardrail visibility,
generated artifact readability, execution-summary boundaries, backlog
readability, and validation module ownership.

## Motivation

The rendered repository now shows the major guardrail fixes working, but reviews
still need a durable place to see raw-check status, generated YAML and Markdown
should stay clean without visual noise, promoted execution journals should not
mirror the exact machine event stream, and validation ownership should continue
moving out of the central orchestration module.

## Actors

- Pipeline maintainers checking whether public raw guardrails are trusted.
- Codex agents reading `execution.md`, `events.yaml`, and `.ai/knowledge`.
- Reviewers maintaining `featurectl_core` validation logic.

## Problem

Public raw correctness is backed by tests and CI, but the current project
knowledge does not expose a compact guardrail status page. `execution.md`
summaries are still closer to event replay than human narrative after
promotion. YAML and Markdown formatting constraints exist, but tests do not yet
lock the no-blank-line YAML style or central status/backlog readability deeply
enough. Review validation helpers still live in `validation.py`.

## Goals

- Add a durable `.ai/knowledge` status document for public raw and CI guardrail
  verification.
- Harden tests so source-controlled generated YAML remains multi-line without
  unnecessary blank lines.
- Make promoted `execution.md` event sections summarize event groups while
  exact records stay in `events.yaml`.
- Split review validation helpers into a focused validator module.
- Improve central backlog formatting and shared knowledge links.

## Non-Goals

- No migration of every historical canonical execution journal.
- No rewrite of the full `featurectl_core` command layer.
- No change to the Native Feature Pipeline skill sequence.
- No new external service or runtime dependency.

## Related Existing Features

- `pipeline/public-raw-artifact-guardrails`
- `pipeline/raw-schema-and-execution-boundary-hardening`
- `pipeline/event-schema-strictness-and-narrative-execution-logs`
- `pipeline/guardrail-polish-and-backlog-tracking`

## Functional Requirements

- FR-001: The repository shall expose a durable guardrail status document that
  records public raw, wrapper execution, compile, and artifact-formatting checks.
- FR-002: Source-controlled generated YAML shall be rejected by tests when it
  contains blank-line noise or collapsed one-line serialization.
- FR-003: Promotion shall rewrite canonical `execution.md` event sections into
  grouped human summaries derived from `events.yaml`.
- FR-004: Detailed event fields shall remain available in `events.yaml` and not
  be duplicated as a second machine event log in `execution.md`.
- FR-005: Review artifact validation shall move from `validation.py` into a
  focused validator module without changing CLI behavior.
- FR-006: Shared knowledge shall link guardrail status, backlog ownership, and
  validation module ownership for future agents.

## Non-Functional Requirements

- NFR-001: Existing `featurectl.py` and `pipelinebench.py` command contracts must
  remain backward compatible.
- NFR-002: Offline tests must not require network access; public raw checks stay
  explicit command or CI checks.
- NFR-003: New Markdown should avoid dense lines over 180 characters outside
  code blocks and tables.
- NFR-004: Validator extraction must avoid circular imports.

## Acceptance Criteria

- AC-001: `.ai/knowledge/pipeline-guardrails-status.md` exists and names the
  raw, wrapper, compile, artifact-formatting, and workflow guardrails.
- AC-002: Artifact-formatting tests reject blank lines in canonical YAML files
  that are generated by the pipeline.
- AC-003: A promoted feature with gate, slice, review, verification, finish, and
  promotion events has a compact `execution.md` event summary.
- AC-004: The same promoted feature preserves exact machine fields in
  `events.yaml`.
- AC-005: `featurectl_core/validators/review.py` owns review validation and the
  full validation suite still passes.
- AC-006: `.ai/knowledge/pipeline-backlog.md`, `module-map.md`,
  `integration-map.md`, and `architecture-overview.md` reference the updated
  status and validator ownership.
- AC-007: Full feature-pipeline tests, public raw checks, compile checks, and
  `git diff --check` pass before promotion.

## Product Risks

- Over-collapsing execution summaries could hide useful context, so exact audit
  data must stay in `events.yaml`.
- A stricter YAML no-blank-line check could flag hand-authored YAML that is not
  generated by the pipeline if the tested path set is too broad.
- Validator extraction could change blocker wording and break downstream tests.

## Assumptions

- The raw-vs-rendered discrepancy reported by external review is stale in a
  fresh environment; this feature preserves regression checks rather than
  changing GitHub raw behavior.
- The current workflow can remain the CI authority for public raw checks on
  `main`.
- Historical canonical features may keep older execution summaries unless a
  migration is explicitly requested later.

## Open Questions

- Should future work publish public raw check results into a badge in
  `README.md`, or is the knowledge status document enough for this repo?
