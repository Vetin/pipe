# Technical Design: Execution Safety Guardrails

## Change Delta

Add transition and gate dependency checks, active event-sidecar requirements,
current-state execution validation, `implementation-ready`, dirty base checkout
handling, richer Docs Consulted parsing, e2e source-diff assertions, partial
fidelity runner metadata, and full deterministic CI coverage.

## Implementation Summary

The implementation keeps `featurectl.py` as a wrapper and changes
`featurectl_core` internals. It adds small helpers for transition legality,
gate dependencies, active workspace event requirements, and docs-consulted
entries. Tests are extended before or with each behavior change.

## Modules And Responsibilities

- `.agents/pipeline-core/scripts/featurectl_core/cli.py`: command parsing and
  mutation commands for `new`, `step set`, `gate set`, `scope-change`,
  `worktree-status`, and `implementation-ready`.
- `.agents/pipeline-core/scripts/featurectl_core/validation.py`: orchestration
  of active workspace validation and Docs Consulted entry checks.
- `.agents/pipeline-core/scripts/featurectl_core/validators/execution_log.py`:
  active current-state and narrative execution log checks.
- `.agents/pipeline-core/scripts/featurectl_core/validators/worktree.py`:
  worktree isolation checks.
- `tests/feature_pipeline/test_real_codex_conversation.py`: opt-in real Codex
  behavioral tests.
- `pipeline-lab/showcases/scripts/run_codex_e2e_case.py`: prompt profile
  fidelity labels.
- `.github/workflows/pipeline-guardrails.yml`: deterministic guardrail and full
  feature pipeline test coverage.

## Dependency And Ownership Plan

Transition and gate dependency helpers are the critical path because they affect
`step set` and `gate set`. Active event-sidecar validation depends on existing
event writer behavior. Worktree/readiness split depends on preserving the old
implementation readiness validation under a new command.

## Contracts

CLI contract additions:

- `featurectl.py new --allow-dirty`
- `featurectl.py implementation-ready --workspace <path>`

CLI behavior changes:

- `featurectl.py step set` rejects illegal jumps.
- `featurectl.py gate set` rejects downstream gate approvals before upstream
  dependencies are satisfied.
- `featurectl.py worktree-status` checks worktree isolation only.

## API/Event/Schema Details

No event schema field changes are required. Existing event types continue to be
used for `step_changed`, `scope_changed`, `gate_status_changed`, and
`run_initialized`. New command failures do not write events.

Outcome-smoke runner manifests add:

```yaml
pipeline_fidelity: partial
not_valid_for_full_pipeline_readiness: true
```

Full native profiles use:

```yaml
pipeline_fidelity: full
not_valid_for_full_pipeline_readiness: false
```

## Core Code Sketches

```python
def validate_step_transition(old_step: str, new_step: str) -> None:
    if old_step == new_step:
        return
    if new_step not in STEP_TRANSITIONS.get(old_step, set()):
        raise FeatureCtlError(
            f"illegal step transition: {old_step} -> {new_step}; "
            "use scope-change for backward replanning"
        )
```

```python
def require_gate_dependencies(state: dict, gate: str, status: str, workspace: Path) -> None:
    if status not in {"approved", "delegated", "complete"}:
        return
    for dependency in GATE_DEPENDENCIES.get(gate, ()):
        if gates.get(dependency) not in {"approved", "delegated", "complete"}:
            raise FeatureCtlError(f"{gate} requires {dependency} approved or delegated")
```

## Data Model

No new workspace file is introduced. Active workspaces must include the existing
`events.yaml` sidecar. Runner reports receive two fidelity metadata fields.

## Error Handling

All new command rejections use `FeatureCtlError`. Test assertions should match
stable substrings such as `illegal step transition`, `requires architecture`,
`events.yaml`, `dirty base checkout`, or `implementation readiness failed`.

## Security Considerations

Under-specified security-sensitive real Codex prompts must not proceed through
full planning without blocking questions. Dirty base checkouts must be explicit
so unrelated user changes are not silently mixed into feature worktrees.

## Test Strategy

- Focused unit tests for step transitions and gate dependencies.
- Validation tests for active missing `events.yaml`, execution state drift, and
  per-path Docs Consulted entries.
- Worktree tests for isolation-only status and separate implementation
  readiness.
- Opt-in real Codex tests for under-specified vs fully specified prompts.
- Runner tests for outcome-smoke partial fidelity metadata.
- CI job for full deterministic `tests/feature_pipeline`.

## Migration Plan

Existing active workspaces generated by the current tool already include
`events.yaml`. Legacy inactive or canonical workspaces without events remain
allowed. Test fixtures that manually create workspaces must add `events.yaml` or
mark themselves legacy.

## Rollback Plan

Rollback individual checks by removing their CLI dependency gate and associated
tests. Keep the feature worktree and evidence to identify which invariant caused
unexpected friction.

## Integration Notes

Manual preflight should call:

```bash
featurectl.py worktree-status --workspace <workspace>
featurectl.py implementation-ready --workspace <workspace>
```

The first checks isolation; the second checks implementation may begin.
